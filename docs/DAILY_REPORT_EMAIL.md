# 日次レポートメール

## サーバー（VPS / cron）で動かす

サーバーにはブラウザがないため、**Google の OAuth（ログイン画面）は自分の Mac（またはブラウザが使えるマシン）で一度だけ**行い、できあがった **`token.json` をサーバーに置く**運用になります。

### 必要なファイル（プロジェクト直下）

| ファイル | 役割 |
|----------|------|
| `credentials.json` | Google Cloud Console で作成した **OAuth 2.0 クライアント ID**（デスクトップアプリ用）の JSON。サーバーにも **同じパス**で置く。 |
| `token.json` | Mac で `send_daily_report.py` を実行してブラウザ認証したあとに自動保存される。**サーバーにも同じ内容を置く。** |

### 毎回 Mac で token を更新する必要があるか？

**いいえ、毎回ではありません。**

- **ふだんの実行（毎朝の cron など）**では、サーバー上の `token.json` に含まれる **リフレッシュトークン**で、アクセス用トークンが自動的に取り直されます（`send_daily_report.py` 内の `creds.refresh`）。**ブラウザは不要**で、Mac に触る必要もありません。
- **Mac でブラウザ認証し直して `token.json` を差し替える**のは、**初回の一度**と、**失効したときだけ**です。失効の例: Google アカウントのパスワード変更、アプリ連携の解除、長期間未使用、セキュリティ上の失効など。頻度は **多くの場合は月〜年単位**で、毎日ではありません（Google のポリシーは変更されることがあります）。

まとめると、**一度サーバーに有効な `token.json` を置ければ、その後は基本は自動**です。

#### 失効の「スパン」は？（目安）

Google は **リフレッシュトークンがいつでも無効化されうる**と書いており、**固定日数で必ず切れる**とは限りません。よくある目安は次のとおりです。

- **アクセストークン**は約 **1 時間**程度で切れるが、**リフレッシュトークンで自動更新**される（cron で毎日動かしていれば問題になりにくい）。
- **OAuth 同意画面が「テスト」のまま**のプロジェクトでは、**約 7 日でリフレッシュが失効**しやすい（個人アプリでよくある原因）。**本番（In production）**にすると長期運用しやすい（スコープによっては検証が必要な場合あり）。
- **長期未使用**（しばしば **6 ヶ月**などと言及されるが、ポリシーは変更され得る）、**パスワード変更**、**アカウントでアプリ連携を削除**、などでも無効になる。

### 手順（初回）

1. **Mac** でリポジトリのルートに移動し、  
   `python3 send_daily_report.py` を実行する。  
2. ブラウザが開くので Google アカウントで許可する。  
3. ルートに **`token.json` が生成・更新**される。  
4. サーバーへ渡す（どちらか）。  
   - **Git で同期**（プライベートリポジトリ想定）: `token.json` をコミットして `push` → VPS で `git pull`（`docs/DEPLOY_GITHUB.md` の「全同期」と同じ考え方）。  
   - **直接コピー**: `scp token.json user@vps:/opt/dpa_app/` など。  
5. サーバーでも **`credentials.json` と `token.json` がプロジェクト直下**にあることを確認する。  
6. **cron** 例（パスは環境に合わせる）:

```cron
0 7 * * * cd /opt/dpa_app && /opt/dpa_app/venv/bin/python send_daily_report.py
```

タイムゾーンはサーバ設定に依存します（日本時間で 7 時にしたい場合はサーバを JST にするか、`cron` の時刻を調整）。

### トークンが失効したとき（`invalid_grant` など）

パスワード変更や長期未使用で **`token.json` の更新に失敗**することがあります。そのときは **もう一度 Mac で** `send_daily_report.py` を実行して再ログインし、**新しい `token.json` をサーバーに上書き**してください（Git `push` でも `scp` でも可）。

### 注意

- `token.json` は **秘密情報**です。パブリックリポジトリにコミットしないでください。  
- サーバーだけで初回認証したい場合は、**Mac で作った `token.json` をコピーする**以外に、OAuth の「リダイレクト URI」やデバイスフローを別途組む必要があり、このプロジェクトのスクリプトは **ローカル認証 → ファイルコピー**を前提にしています。

## メール送信の ON/OFF

設定画面（メール通知）で「毎朝メールを送る」のトグルで ON/OFF できます。  
OFF の場合、cron で `send_daily_report.py` 実行時も **バッチは実行**しますが、**メール送信はスキップ**されます。  
（config.yaml の `daily_report.enabled: false` でも同様）

## 本文例

`send_daily_report.py` で送信されるメールは **HTML** 形式です。  
**保有銘柄の状況**と**ウォッチリスト優先度**は**表（テーブル）**で表示されます。

---

## 件名

```
DPA 日次レポート 2026-03-15
```

（データ日が入ります。無い場合は「DPA 日次レポート」のみ。）

---

## 本文の構成（HTML）

- 冒頭：バッチ状態・完了日時
- **本日の目標現金比率**・**マクロフェーズ**（VI Z・MACD）
- **ポートフォリオ概要**（総資産・現金・株式評価額・新規購入枠）
- **売却指示**（なし or 表）
- **保有銘柄の状況** → **表**（銘柄・名前・現在%・目標%（乖離%pt）・スコア・株価）
- **ウォッチリスト優先度（保有すべき順）** → **表**（順位・銘柄・名前・状態・スコア・株価）
- **新規購入推奨**（なし or 表）

メールクライアント（Gmail 等）では表が罫線付きで表示され、アプリを開かずに一覧で確認できます。
